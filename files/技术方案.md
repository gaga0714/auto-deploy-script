# 一键部署脚本 · 技术方案

## 1. 背景与目标

`脚本.md` 描述的需求是「在服务器上一键完成新项目的 nginx 接入」,涉及五件事:

1. 把构建产物放到 `/home/html/<project>` 下
2. 生成 `/etc/nginx/sites-available/<project>` 配置(基于固定模板)
3. **自动分配端口**(查询当前已占用端口,取最大值 +1)
4. 软链到 `sites-enabled`
5. `nginx -t` 检查,通过则 `systemctl restart nginx`

本方案给出一个**幂等、可回滚、与现有 CI 解耦**的实现设计,目标:

- 同一项目重复执行不会出错(已存在的 config 复用而非覆盖)
- 任何一步失败可自动回滚到上一致状态,不留半成品
- 与本仓库已有的 GitHub Actions 部署流程(`.github/workflows/deploy.yml`)互补:CI 负责**增量同步文件**,本脚本负责**首次接入 / 端口分配 / nginx 配置**

## 2. 范围划分

| 关注点 | 由谁负责 | 触发时机 |
|---|---|---|
| 文件构建与同步 | 现有 GitHub Actions(`flutter build web` + `rsync`) | 每次 push 到 `main` |
| nginx 接入(首次配置) | 本方案的 `setup-site.sh` | 新项目首次上线时手动跑一次 |
| 端口分配 | `setup-site.sh` 内部子步骤 | 同上 |
| nginx reload | `setup-site.sh` 末尾 | 同上;后续 CI 不动 nginx |

**重要边界**:CI 只 rsync 文件,**不碰 nginx**。一旦 nginx 配置就位且端口固定,后续部署完全无需 reload(静态文件直接被 nginx 读取)。

## 3. 总体架构

```
开发者本机 / CI
      │
      │  git push main
      ▼
GitHub Actions: flutter build web → rsync 到 /home/html/<project>/
      │
      │
      │  (新项目第一次上线时)
开发者 ssh 到服务器,手动执行一次:
      ▼
   setup-site.sh <project-name>
      ├─ 1. 校验 /home/html/<project> 存在
      ├─ 2. 扫描已有 listen 端口,计算 next_port
      ├─ 3. 渲染 nginx 配置模板
      ├─ 4. 写入 sites-available/ → ln 到 sites-enabled/
      ├─ 5. nginx -t
      └─ 6. systemctl reload nginx  (失败则回滚 3~4 步)
```

## 4. 详细设计

### 4.1 脚本接口

```
setup-site.sh <project-name> [--root <path>] [--port <port>]
```

| 参数 | 说明 | 默认 |
|---|---|---|
| `<project-name>` | 必填,用作 config 文件名、`server_name` 后缀 | — |
| `--root` | 静态文件根目录 | `/home/html/<project-name>` |
| `--port` | 手动指定端口,跳过自动分配 | 自动分配 |

返回码:`0` 成功,非 0 表示某步骤失败(消息打到 stderr)。

### 4.2 端口分配算法

**数据源**:扫描 `/etc/nginx/sites-available/*` 与 `/etc/nginx/sites-enabled/*` 中所有 `listen` 指令(去重),取最大值 +1。

**为什么用 sites-* 而不是 `ss -tlnp`**:
- `ss` 反映「当前进程监听的」,nginx reload 前后会有差;config 文件是「**意图**」,跨重启稳定
- 即便端口当下没在监听(比如 nginx 没起),也不会撞号

**伪代码**:

```bash
next_port() {
  local max
  max=$(grep -rhoE 'listen[[:space:]]+[0-9]+' \
        /etc/nginx/sites-available /etc/nginx/sites-enabled 2>/dev/null \
        | awk '{print $2}' \
        | sort -un \
        | tail -1)
  echo $(( ${max:-8079} + 1 ))   # 起始基线 8080
}
```

**边界**:
- 没有任何 site 存在 → 起始 `8080`(常见 web 端口区间)
- 已分配端口里有 `80`/`443` → 不影响算法(它们也算 max 候选,但 +1 后是 `81`/`444`,与 `8080+` 不冲突;若想强制业务端口在高位,可在 awk 里过滤 `< 1024`)

### 4.3 nginx 配置模板

```nginx
# /etc/nginx/sites-available/<project-name>
server {
    listen __PORT__;
    server_name localhost;

    root __ROOT__;
    index index.html;

    # Flutter web SPA:任何路径回退到 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 静态资源缓存策略(Flutter 输出文件名带 hash,可激进缓存)
    location ~* \.(js|css|woff2?|png|jpg|svg|webp)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # index.html 不缓存,保证更新立即生效
    location = /index.html {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }
}
```

**与 `脚本.md` 原模板的两处增强**:
1. `try_files` 末尾加 `/index.html`:Flutter web 用 hash 路由,刷新子路径不会 404
2. 加入 SPA 友好的缓存策略:hash 资源长期缓存,`index.html` 不缓存,符合 Flutter web 的指纹规则

### 4.4 幂等与回滚

| 场景 | 行为 |
|---|---|
| 同名 config 已存在 | 跳过写入,打印「已存在,复用端口 N」 |
| 软链已存在但指向不同文件 | 报错退出,提示人工介入(不自动改) |
| `nginx -t` 失败 | 删除新写入的 config + 软链,**不 reload**,退出码 1 |
| `systemctl reload nginx` 失败 | 同上(虽极少发生,test 通过 reload 应能过) |

回滚通过先「写临时文件 + 临时软链」、test 成功后再 mv 到正式名落地的方式实现 —— 避免破坏现有运行配置。

### 4.5 权限模型

- 脚本必须以 `sudo` 或 `root` 身份运行(读写 `/etc/nginx/`、调 `systemctl`)
- `/home/html/<project>` 的 owner 是 CI 的部署用户(本仓库已是),与脚本运行用户解耦

### 4.6 完整伪代码

```bash
#!/usr/bin/env bash
set -euo pipefail

PROJECT="${1:?usage: setup-site.sh <project-name> [--root path] [--port N]}"
ROOT="/home/html/$PROJECT"
PORT=""

# 解析可选参数
shift
while [[ $# -gt 0 ]]; do
  case "$1" in
    --root)  ROOT="$2"; shift 2;;
    --port)  PORT="$2"; shift 2;;
    *) echo "unknown arg: $1" >&2; exit 2;;
  esac
done

AVAIL="/etc/nginx/sites-available/$PROJECT"
ENABLED="/etc/nginx/sites-enabled/$PROJECT"

# 1. 校验
[[ -d "$ROOT" ]] || { echo "root not found: $ROOT" >&2; exit 1; }

# 2. 幂等:已存在则复用
if [[ -e "$AVAIL" ]]; then
  echo "config exists, reusing: $AVAIL"
  exit 0
fi

# 3. 端口分配
if [[ -z "$PORT" ]]; then
  PORT=$(next_port)   # 见 4.2
fi

# 4. 渲染 + 落盘到 .tmp
TMP="${AVAIL}.tmp"
sed -e "s|__PORT__|$PORT|g" -e "s|__ROOT__|$ROOT|g" \
    /usr/local/share/menu-play/nginx-site.tpl > "$TMP"

# 5. 临时符号链接 + 测试
ln -sf "$TMP" "$ENABLED"
if ! nginx -t; then
  rm -f "$ENABLED" "$TMP"
  echo "nginx -t failed, rolled back" >&2
  exit 1
fi

# 6. 正式落地
mv "$TMP" "$AVAIL"
ln -sfn "$AVAIL" "$ENABLED"
systemctl reload nginx

echo "ok: $PROJECT listening on :$PORT, root=$ROOT"
```

## 5. 与 CI 的集成

**默认不集成**。新项目首次手动跑一次,后续 CI 只管 rsync 文件,nginx 不需要重启。

如果未来希望「全自动」(本仓库当前不必),可以在 `.github/workflows/deploy.yml` 里加一步:

```yaml
- name: Ensure nginx site
  run: |
    ssh -p $SSH_PORT $SSH_USER@$SSH_HOST \
      "sudo /usr/local/bin/setup-site.sh menu-play"
```

需要的前置条件:
- 脚本固化到服务器的 `/usr/local/bin/`
- `SSH_USER` 拥有 `setup-site.sh` 的免密 `sudo`(通过 `/etc/sudoers.d/`)
- 脚本本身幂等(已设计为如此)

## 6. 安全考虑

| 风险 | 缓解 |
|---|---|
| 任意写 nginx config | 脚本走 sudo,owner=root,普通用户无写权限 |
| 端口扫描泄漏内部布局 | nginx `server_name` 固定为 `localhost`,通过外层反代/防火墙暴露 |
| `nginx -t` 通过但运行时崩 | reload 失败时手动介入(reload 不会中断现有连接) |
